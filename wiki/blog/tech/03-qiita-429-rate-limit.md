# 親子対話：Qiita の 429（レート制限）に遭遇した話

父「また赤い文字が出た」

娘「どれどれ」

父「`FAIL: 429` って出てるんだけど」

娘「あー、Qiita に怒られたね」

父「何が悪かったんだ」

娘「投稿しすぎ。Qiita が『ちょっと待って』って言ってる」

父「6 記事を一気に公開しようとしただけなんだけど」

娘「その『だけ』がダメなの」

父「だって 10 秒間隔で開けたぞ」

娘「Qiita のレート制限はね、短時間の連続投稿にはもっと厳しいのよ」

父「具体的にどんな制限なんだ」

娘「公式には『1 時間 1000 リクエスト』って書いてある」

```
https://qiita.com/api/v2/docs#%E3%83%AC%E3%83%BC%E3%83%88%E5%88%B6%E9%99%90
```

父「1000 回もできるなら余裕じゃないか」

娘「それがそうでもない。POST と PATCH には別途制限があるみたい」

父「別途？」

娘「短時間に連続で PATCH すると、1000 回に達する前にブロックされる」

父「つまり『1 時間 1000 回』っていう全体制限とは別に、『投稿系は○分で○回まで』って制限があるってこと？」

娘「そう。Qiita のエラーメッセージにも書いてあったろ」

```
サービスの安定運用のため、一定時間内の投稿数を制限させていただいております。
下書き保存の上、しばらく時間をおいてから再度お試しください。
```

父「たしかに。『一定時間内の投稿数』って書いてある」

娘「POST（新規投稿）と PATCH（更新）が対象で、GET は大丈夫」

父「GET は 6 件全部成功したわ」

娘「でしょ。実際にブロックされたのは PATCH のところ」

父「じゃあどうすればいいんだ」

娘「いくつか方法がある」

## 対策 1：時間を空ける

父「一番シンプルなやつ」

娘「PATCH の間隔を 30 秒とか 60 秒に伸ばす」

```
❌ 10 秒間隔 → 429
✅ 60 秒間隔 → OK
```

父「じゃあ 60 秒にすればいいのか」

娘「それでも 6 記事連続だと引っかかるかも。余裕を見て 2〜3 分くらいが安全」

父「6 記事 × 3 分 = 18 分か」

娘「余裕があったらそれくらい見といたほうがいい」

## 対策 2：時間帯を変える

父「混んでる時間と空いてる時間がある？」

娘「Qiita は日本のサービスだから、日本の昼間は混み合う」

父「じゃあ夜中とか早朝が狙い目？」

娘「そう。特に早朝 4〜6 時は空いてることが多い」

## 対策 3：下書きにしてからまとめて公開しない

父「今やってるのがこれなんだけどな」

娘「『下書きで投稿 → 後で PATCH で公開』は便利なんだけど、PATCH が連続するとブロックされる」

父「じゃあ最初から公開で投稿すればいいのか」

娘「それも手。1 記事ずつ POST すれば、PATCH の制限は関係なくなる」

父「でも連続 POST も制限されない？」

娘「POST にも制限はあるけど、PATCH よりは緩い気がする。体感だけど」

## 対策 4：リトライ処理を入れる

父「429 が来たら自動で待つってこと？」

娘「そう。`Retry-After` ヘッダーを見て、指定秒数待ってから再送する」

```javascript
async function publishWithRetry(id, retries = 3) {
  for (let i = 0; i < retries; i++) {
    const res = await fetch(`https://qiita.com/api/v2/items/${id}`, {
      method: 'PATCH', headers,
      body: JSON.stringify({ private: false })
    });
    if (res.ok) return;
    if (res.status === 429) {
      const wait = 120_000; // 最低2分待つ
      console.log(`429 → ${wait / 1000}秒待機...`);
      await new Promise(r => setTimeout(r, wait));
    }
  }
}
```

父「2 分待ってダメならさらに待つ？」

娘「リトライ回数を決めておいて、全部ダメなら手動でやる」

## 対策 5：事前に全記事を下書き投稿しておいて、手動で公開

父「一番確実？」

娘「そう。下書き投稿（POST）は 1 回で終わるから制限にかかりにくい」

父「公開は Qiita の管理画面からポチポチやる」

娘「CLI にこだわらなければこれが一番安全」

父「でもそれだと自動化の意味が…」

娘「そこはトレードオフ。レート制限との付き合い方次第」

## Qiita のレート制限まとめ

父「つまりこういうことか」

| 操作 | 制限の厳しさ | 備考 |
|------|-------------|------|
| GET | 緩い | 1000 req/h の枠内 |
| POST | 中程度 | 新規投稿は比較的通りやすい |
| PATCH | 厳しい | 短時間連続はブロックされやすい |

娘「そういうこと。今回引っかかったのは PATCH の連続」

父「じゃあ対策として」

娘「そうね」

1. **PATCH の間隔は最低 60 秒、できれば 2〜3 分**
2. **時間帯を選ぶ（早朝がベスト）**
3. **初回から公開で投稿するのも手**
4. **429 が来たらリトライ処理で自動待機**
5. **どうしてもダメなら手動公開**

父「今回の _publish_slow.mjs はどうするんだ」

娘「60 秒間隔に修正して、時間を置いてから再実行する」

父「あとで投稿する用に変更しておくか」

娘「それがいいね。Qiita に優しくしよう」

父「というわけで、今回は Qiita に優しくすることにした」

娘「待てば海路の日和あり。429 は『あとでね』って言ってるだけ」

父「なるほど。焦らず行こう」

娘「レート制限はサーバーからの『ありがとう、ちょっと休んでね』というシグナル。ちゃんと聞こうね」
